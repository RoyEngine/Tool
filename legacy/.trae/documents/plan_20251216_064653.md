# 恢复项目规则中的文件结构规范计划

## 问题分析
之前修改project_rules.md文件时，为了使其更符合"定义TRAE对话规则"的核心目标，移除了一些项目开发规范相关的内容，包括明确的文件结构规则。

## 优化方案
将文件结构规则重新添加回project_rules.md文件中，同时保持文件的整体结构和目的。

## 修改内容

### 1. 在现有规则中添加文件结构规范
在"项目特定规则"部分添加明确的文件结构规则，包括：
- 源代码文件的存放位置
- 配置文件的存放位置
- 测试文件的存放位置

### 2. 保持文件的核心目的
确保文件仍然以"定义TRAE在当前项目中对话时需要遵循的规则"为核心，文件结构规范作为项目特定规则的一部分。

### 3. 优化规则表达方式
使用清晰、直接的语言，采用列表形式，易于阅读和理解。

## 修改后的规则格式
```markdown
# 项目规则
定义TRAE在当前项目中对话时需要遵循的规则。

## 1. 项目特定规则
- 了解并遵循项目的核心功能和架构
- 熟悉项目的文件结构和命名规范：
  - 源代码文件放在`src/`文件夹
  - 配置文件放在`config/`文件夹
  - 测试文件放在项目根目录
- 关注项目的本地化功能和相关模块

## 2. 对话要求
- 全程使用中文
- 输出内容简洁、清晰
- 针对项目相关问题提供专业、准确的回答

## 3. 代码生成规范
- 严格遵循项目的代码风格和规范
- 生成的代码符合项目的文件结构
- 优先使用项目已有的模块和函数

## 4. 本地化相关规则
- 了解项目的本地化流程和工具
- 关注本地化文件的格式和结构
- 提供本地化相关问题的专业解决方案

## 5. 版本控制
- 了解项目的Git工作流和分支管理
- 提供符合项目版本控制规范的建议

## 6. 依赖管理
- 了解项目的依赖管理策略
- 提供符合项目依赖规范的建议
```

## 预期成果
修改后的project_rules.md文件将同时包含：
1. TRAE对话规则
2. 明确的项目文件结构规范
3. 其他项目特定规则

这样可以确保TRAE在对话时既遵循对话规则，又了解项目的文件结构规范，提供更准确、专业的回答和代码生成。
